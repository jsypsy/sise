## 3. ⚠️ 가장 중요한 리스크 — 시그널 연속성 단절
개편으로 단지의 `sgg_cd`가 **옛코드 → 새코드**로 바뀌면:
- 같은 단지의 과거 거래는 **옛 sgg_cd**, 7.1 이후 거래는 **새 sgg_cd** 로 저장됨.
- 시그널 그룹이 (apt_nm, **sgg_cd**, pyeong)이므로 **두 그룹으로 쪼개짐** → 새코드 그룹엔 직전·전고점 이력이 없어 **모든 거래가 거짓 '신고가'로 오탐**.
- CLAUDE.md 불변 규칙("전고점·직전가는 동일 그룹 이전 거래로", "신고가 오탐 없음") 위반.

### 대응 방향 (택1, 처리 시 결정)
- **(A) 권장: 옛→새 sgg_cd 정규화 맵** 도입. ingest 단계 또는 `signals_v`에서 옛코드를 **새 canonical 코드로 매핑**해 같은 단지가 한 그룹으로 유지되게. (예: 28110/28140 → 제물포구 새코드)
  - 단, 옛코드 시기의 행정구역과 새코드 구역 경계가 1:1이 아닐 수 있어(중구가 제물포구·영종구로 분할) **법정동(읍면동) 단위 매핑이 필요할 수 있음** → 단순 sgg 단위 매핑으로 부족하면 umd_nm까지 봐야 함.
- (B) 차선: 전환기 동안 해당 지역 신고가 판정을 보수적으로(또는 표시 보류). 데이터는 모으되 오탐만 억제.

## 4. API 중단창(6.30 18:00~7.1 09:00) 처리
- 7.1 KST 05:10 cron(= UTC 20:10)은 **중단창 안**이라 인천·광주·전남 조회가 빈 응답/실패일 수 있음.
- ingest는 이미 재시도+에러 로깅으로 **크래시 없이 건너뜀** → 그날 해당 지역 데이터만 누락.
- **자가복구**: ingest는 당월을 매일 재수집하므로, 7.2 cron(09:00 이후)에서 7.1 데이터가 backfill됨. → 윈도우 자체는 큰 조치 불필요. **단, 7.1~7.2 수집 로그 확인**.
