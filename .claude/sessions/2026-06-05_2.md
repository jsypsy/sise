## 의사결정 기록

### 채택한 결정

1. **R2 CORS 설정 추가 (Cloudflare 대시보드)**
   - 이유: 브라우저에서 R2 fetch 시 `Access-Control-Allow-Origin` 헤더 없어 차단. 서버사이드 fetch는 정상.
   - 조치: Cloudflare R2 버킷 → Settings → CORS Policy에 `AllowedOrigins: ["https://sise-ten.vercel.app"]` 추가.

2. **TrendChart를 fetch-free pure display component로 전환**
   - 이유: R2 데이터를 ComplexInner와 TrendChart가 이중 fetch했음. `deals` prop으로 통일해 요청 1회로 줄임.
   - 부수효과: 기간 필터를 props 없이 내부 state로 관리 가능해져 설계가 단순해짐.

3. **동 목록을 클라이언트에서 aptList로 파생**
   - 이유: 법정동(MOLIT) 기준 `umd_nm`은 사용자가 인지하는 행정동과 불일치(고덕아이파크 → 상일동). 별도 API 호출로 동 목록을 가져오면 단지 누락 발생.
   - 해결: `?sgg=` 한 번에 시군구 전체 단지 fetch → `umd_nm` 기준으로 클라이언트 동 목록 파생 → 동 필터링도 클라이언트에서.
   - 부수효과: API 호출 횟수 감소 (시군구 선택 시 1회로 동+단지 모두 해결).

4. **평형별 최고가는 `computeDeltas` 내에서 계산 (별도 쿼리 없음)**
   - 이유: R2 raw deals 데이터를 이미 보유하므로 별도 Supabase 쿼리 없이 in-memory 계산 가능.
   - 로직: 먼저 취소 제외 deals를 스캔해 `peakByPy: Map<py, maxPrice>` 구성 → 각 거래에 `is_high = (price === peak)` 플래그.

### 검토했지만 채택하지 않은 대안

| 대안 | 기각 이유 |
|---|---|
| 동 목록을 별도 `/api/umd?sgg=` API로 분리 | 추가 Supabase egress + 별도 fetch. aptList에서 파생하면 공짜. |
| 법정동 → 행정동 매핑 테이블 도입 | 구현 복잡도 높음, 유지보수 부담. 현재 단계에서 불필요. |
| 동+단지 합쳐서 `단지명 · 동명` 포맷 단일 드롭다운 | 옵션 가독성이 떨어지고 동 필터링 기능 없어 목록이 길어짐. 사용자가 분리 요청. |
| `signals_mv` 기반 최고가 배지 | R2 전환 이후 signals_mv를 complex 페이지에서 조회 않기로 결정됨. R2 데이터에서 계산하는 게 일관적. |
