**문제:** 신축 입주장 단지는 소유권이전 전 거래가 매매 API가 아니라 **분양권전매 API(입주권/분양권)**에 들어감 → 매매만 보면 전고점 저평가(올림픽파크포레온 84타입 매매 31.3억 vs 입주권 33억 = 아실 최고가).
- **진단 워크플로우 신설:** `inspect-complex`(R2+DB), `inspect-molit`(매매 원본 유형), `inspect-presale`(분양권 API 스키마·권한), `probe-region`(코드 생사).
- 확인: 아실 33억 = **입주권**(2025-10-16, `ownershipGbn=입`). MOLIT 분양권 API(`RTMSDataSvcSilvTrade`) 활용신청 후 접근(같은 계정 키, 전파 지연 겪음).
- **구현:**
  - `lib/refine.ts` `refineSilvItem` — 분양권 스키마(ownershipGbn 입/분→trade_type, aptSeq/buildYear/roadNm 없음). raw_key에 trade_type 접두로 매매와 충돌 방지. refineItem은 trade_type='매매' 태깅.
  - `0020_trade_type.sql` — transactions에 `trade_type` 컬럼 + signals_mv 재생성(SELECT * 로 자동 흐름). 입주권 중개거래가 기존 WHERE(중개거래·취소제외) 통과 → **전고점/신고가 자동 통합(아실식)**.
  - `scripts/ingest.ts` — 분양권 루프 추가(별도 API·별도 quota, 매매와 격리). fetchPage에 endpoint 인자.
  - `scripts/fetch_peaks.ts` — dual-API 크롤(매매+분양권)로 과거 입주권도 hp/R2에 backfill. RawDeal에 tt.
- **검증:** ingest dual-API 실행 → 분양권 **5,154건 upserted, 오류 0**. 올림픽 202510 backfill로 33억 전고점 즉시 교정.
