## 의사결정 기록

### 채택한 결정

1. **`raw_key` 변경 없이 `apt_seq`를 별도 컬럼으로 추가**
   - 이유: `raw_key` 변경 시 기존 적재 데이터와 중복 재적재 발생. `apt_seq`는 단지 식별용이지 거래 유일성 키가 아님.
   - `apt_seq`는 향후 `/complex` 단지 검색 정확도 개선에 활용 예정.

2. **resultCode=22를 `throw new Error("429")`로 처리**
   - 이유: HTTP 200이지만 실질적으로 일일 quota 소진. 기존 `consecutive429` 카운터 로직을 그대로 재활용해 조기 종료 트리거.
   - 별도 에러 타입 만들지 않음 — 현재 코드 구조상 "429" 문자열 포함 여부로 판단하는 곳과 일관성 유지.

3. **`fetch_peaks.ts`의 resultCode=22에서 `return` (throw 아님)**
   - 이유: `fetch_peaks.ts`는 장시간 배치 스크립트. throw하면 `Promise.all` 배치 전체가 reject되고 누적 peaks 데이터 손실.
   - `return`으로 해당 지역만 중단하고 나머지 지역 + 최종 upsert는 정상 완료.

4. **`historical_peaks` + Storage `trends/` 리셋**
   - 이유: bootstrap의 불완전한 fetch-peaks로 `historical_peaks` 데이터가 이미 부정확한 상태. 클린 재구축이 GREATEST 전략 위에서 덮어쓰는 것보다 명확.
   - `transactions`는 rolling window이므로 리셋 불필요.

### 검토했지만 채택하지 않은 대안

| 대안 | 기각 이유 |
|---|---|
| `raw_key`에 `aptSeq` 포함 | 기존 데이터 중복 재적재 발생. 거래 유일성 키로 `aptSeq` 단독 사용 불가 (동일 거래의 재신고 케이스 존재) |
| resultCode=22에서 fetch_peaks.ts도 throw | `Promise.all` 배치 reject → 누적 peaks 데이터 손실 위험. return으로 충분 |
| `transactions`도 truncate | rolling window로 자동 만료. 리셋 불필요하고 당일 quota 소진으로 즉시 재수집 불가 |
| SQL로 Storage 파일 삭제 | Supabase가 직접 SQL 삭제를 차단 (`protect_delete` 트리거). Storage API 사용 필요 |
