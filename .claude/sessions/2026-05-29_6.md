## 의사결정 기록

### 채택한 결정

1. **signals_v → signals_mv (구체화 뷰)**
   - 이유: 일반 뷰는 쿼리마다 전체 테이블 윈도우 함수 재계산 → Supabase free tier 타임아웃 → 빈 화면. 구체화 뷰로 결과를 디스크에 저장하고 ingest 후 1회만 갱신.
   - 기존 `refresh_signals_mv()` SECURITY DEFINER 함수로 ingest.ts에서 RPC 호출.

2. **날짜 드롭다운: get_deal_dates() RPC**
   - 이유: `LIMIT 300` 후 JS dedup은 하루 1,200건 데이터에서 1~2일치만 반환. DB에서 `SELECT DISTINCT deal_date` 직접 실행이 올바른 방법.
   - `RETURNS SETOF date` 대신 `RETURNS TABLE(deal_date text)`로 명시적 컬럼명 부여 → JS에서 `d.deal_date`로 올바르게 매핑.

3. **historical_peaks 스냅샷 방식 (신고가 정확도)**
   - 배경: 현재 DB에 2025-01 이후 데이터만 있어 그 이전 전고점(특히 2022년 피크) 미반영 → 신고가 오탐.
   - 원본 백필(2022~2024)은 413MB + ~730MB 추가 → 500MB 한도 초과로 불가.
   - 채택: 그룹별 최고가 1행만 저장 → ~20~30MB. signals_mv에서 `GREATEST(DB 내 전고점, 과거 전고점)`으로 보정.
   - NULLIF(GREATEST(COALESCE(window, 0), COALESCE(hp.peak_price, 0)), 0)으로 양쪽 NULL 케이스 안전 처리.

4. **날짜 드롭다운 범위: 최근 20 영업일**
   - apt2.me가 최근 20 영업일 기준임을 확인. 거래일은 주말/공휴일 거래가 거의 없어 get_deal_dates(20)이 실질적으로 20 영업일 = 약 1개월.

### 검토했지만 채택하지 않은 대안

| 대안 | 기각 이유 |
|---|---|
| 2022~2024 원본 전체 백필 | DB 413MB + 730MB 추가 = 1,143MB → 500MB 한도 대폭 초과 |
| Supabase Pro 업그레이드 | $25/월 비용 발생. 광고 수익 전 지출 불가 |
| UI 고지문만 추가 | 신고가 오탐이 핵심 가치 훼손. 근본 해결 필요 |
| RETURNS SETOF date 유지 | PostgREST가 `{get_deal_dates: "..."}` 형태로 반환 → JS에서 `d as string` 오동작 |
