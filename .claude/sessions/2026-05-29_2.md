## 의사결정 기록

### 채택한 결정

1. **`lib/digest.ts` 공유 함수 추출**
   - 이유: `app/digest/page.tsx`(서버 컴포넌트)와 `app/api/digest/route.ts` 모두 동일한 텍스트 생성 로직이 필요. 중복 10줄 이상이므로 공유 함수로 추출.
   - CLAUDE.md "Simplicity First" 기준: 10줄 이상 중복이면 추상화가 정당화됨.

2. **`useSearchParams` + `<Suspense>` 패턴으로 ISR 유지**
   - 문제: 지역 필터를 서버에서 처리하면 `searchParams` 접근으로 ISR이 깨짐 (dynamic rendering 강제).
   - 결정: 서버 컴포넌트(`page.tsx`)가 전체 데이터를 ISR 정적 생성으로 가져오고, `TodayClient`(클라이언트 컴포넌트)가 URL searchParams를 읽어 클라이언트 측에서 필터링.
   - `<Suspense>`로 감싸서 Next.js의 강제 dynamic 렌더링 방지.
   - 결과: `/`가 `1d` revalidate 정적 페이지로 빌드됨 ✅

3. **`/api/search`를 검색과 이력 조회 겸용 엔드포인트로 설계**
   - `?q=단지명` → LIKE 검색 + JS 집계
   - `?apt=단지명&sgg=코드` → signals_v 이력 조회
   - 이유: CLAUDE.md Key Files 목록에 `/api/search`만 명시되어 있어 별도 `/api/complex` 추가를 최소화.

4. **검색 결과 집계를 JS에서 처리 (Postgres GROUP BY 대신)**
   - 이유: Supabase JS에서 GROUP BY를 사용하려면 `.rpc()` 호출이 필요하고 새 DB 함수(마이그레이션)가 필요함. 검색 결과는 최대 300행이라 JS 루프로 충분히 처리 가능.
   - CLAUDE.md "시그널/집계는 Postgres에서" 원칙은 *시그널 계산*에 한정 적용. 검색 집계는 예외.

5. **`scripts/ingest.ts` 타입 수정: `filter(Boolean)` → 타입 프리디케이트**
   - 이유: TypeScript 빌드 오류. `filter(Boolean)`은 TS에서 `null`을 제거해도 `(T | null)[]`로 타입이 남음.
   - `(r): r is NonNullable<typeof r> => r != null` 패턴으로 타입 안전하게 수정.

### 검토했지만 채택하지 않은 대안

| 대안 | 기각 이유 |
|---|---|
| 서버에서 searchParams 읽어 DB 필터링 | `export const revalidate` + searchParams 공존 불가 → ISR 깨짐, 불변 아키텍처 결정 #3 위반 |
| 클라이언트에서 Supabase 직접 쿼리 | CLAUDE.md 금지사항: "클라이언트에서 Supabase로 직접 대량 쿼리 금지" |
| `/api/complex` 별도 Route Handler 추가 | CLAUDE.md Key Files에 없음. `/api/search` 겸용으로 충분히 해결 가능 |
| Postgres RPC 함수로 검색 집계 | 새 마이그레이션 필요, 검색 300행 JS 집계로 충분. 복잡성 증가 대비 이득 없음 |
