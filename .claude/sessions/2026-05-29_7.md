## 의사결정 기록

### 채택한 결정

1. **transactions rolling 20일 + historical_peaks 분리**
   - 이유: transactions 413MB로 500MB 한도 근접. rolling window로 용량 유지.
   - historical_peaks(~20MB)이 신고가 판정의 핵심. 월별 차트용 데이터는 Storage(1GB 무료).

2. **월별 추이 데이터 → Supabase Storage 파일**
   - 이유: 월별 그룹×월 전체 기록 시 DB ~420~720MB 초과 위험. Storage 1GB 무료 한도 내 (~200MB).
   - SQL이 Storage를 읽을 수 없으므로 historical_peaks(DB)는 그대로 유지 필수.

3. **fetch_peaks.ts에서 historical_peaks + Storage 동시 처리**
   - 이유: API 순회가 가장 비싼 자원. 같은 순회에서 두 목적지에 동시 저장하면 추가 API 호출 없음.

4. **bootstrap 워크플로우 (1회성 자동화)**
   - 이유: truncate → ingest → fetch-peaks를 수동으로 순서대로 실행하는 것은 오류 가능성 높음.
   - GitHub Actions 버튼 1번으로 전체 플로우 자동화.

5. **ingest 순서: sync → archive → refresh**
   - sync: 신규 거래를 historical_peaks에 즉시 반영 (오늘 신고가 기록 즉시 저장)
   - archive: sync 후 삭제하므로 peaks 유실 없음
   - refresh: 정제된 데이터로 signals_mv 갱신

### 검토했지만 채택하지 않은 대안

| 대안 | 기각 이유 |
|---|---|
| 월별 추이 DB 저장 | ~420~720MB 예상 → 500MB 한도 초과 위험 |
| Storage 파일로 historical_peaks 대체 | SQL(signals_mv)이 Storage 읽기 불가. 아키텍처적으로 불가능 |
| ingest에서만 Storage 파일 업데이트 | 현재 달만 갱신 가능. 과거 이력은 fetch-peaks가 필요 |
| 현재 fetch-peaks 유지하고 Storage 별도 스크립트로 | API 이중 순회. fetch-peaks에서 동시 처리가 효율적 |
