## 의사결정 기록

### 채택한 결정

1. **차트 단일 라인으로 단순화**
   - 이유: 평형이 여러 개일 때 멀티라인 차트는 가독성이 낮음. 평형 버튼이 이미 있으므로 버튼 선택 → 차트 반영이 더 직관적.
   - 구현: `aggregateSingle()` 함수로 선택된 평형(또는 전체)의 월별 최고가만 1줄로 집계.

2. **전체 데이터 초기화 후 재적재**
   - 이유: 고덕아이파크가 DB/R2 어디에도 없음을 확인. 세션 메모의 "상일동" 표기도 오류(실제 고덕동). MOLIT API에서 fetch-peaks가 강동구를 처리했으나 해당 단지가 포함되지 않은 원인 불명. 완전 재적재로 클린 스테이트에서 재확인.
   - 판단: 고덕아이파크가 MOLIT 데이터에 없는 거라면 재실행해도 결과 동일. 하지만 현재 데이터 무결성 확인 차원에서 전체 재적재가 합리적.

3. **샤드 1~5 재적재를 기존 fetch-peaks-schedule.yml에 날짜 추가로 해결**
   - 이유: 별도 워크플로 불필요. 기존 날짜 매핑 방식 그대로 UTC 06-08~12에 샤드 1~5를 매핑하면 충분.
   - 비용: MOLIT quota 하루 1샤드 × 5일 → 기존 설계 범위 내.

### 검토했지만 채택하지 않은 대안

| 대안 | 기각 이유 |
|---|---|
| 샤드 1~5 수동 dispatch | 사용자가 매일 GitHub에서 버튼 눌러야 함. 번거롭고 누락 가능성 |
| reset 직후 shard 1 즉시 실행 | 샤드 6이 같은 날 밤 자동 실행 → MOLIT quota 하루 2샤드 = ~18,000콜 초과 |
| 별도 retry-peaks.yml 워크플로 생성 | 기존 fetch-peaks-schedule.yml에 날짜 추가만으로 충분 — 불필요한 파일 생성 회피 |
