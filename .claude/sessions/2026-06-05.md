## 의사결정 기록

### 채택한 결정

1. **거래 이력을 R2 raw 데이터 기반으로 완전 교체**
   - 이유: `signals_mv`는 Supabase `transactions` 롤링 윈도우 기반 → 최근 ingest 결과만 노출.
     래미안힐스테이트고덕처럼 최근 신고가 없는 단지는 2건만 보임.
     R2에 2006~현재 전체 이력이 있으므로 여기서 읽는 게 올바름.
   - 부작용: `is_high`/`is_rebound` 배지 제거됨. 현재 단지 이력에서는 시그널 배지보다 전체 이력 확인이 더 중요하다고 판단.

2. **월별 집계를 클라이언트에서 계산 (서버 집계 파일 없음)**
   - `fetch_peaks.ts` 설계 의도 그대로 유지: "차트는 클라이언트가 raw에서 직접 계산"
   - 별도 집계 파일 생성 불필요 → R2 파일 1종류로 단순화

3. **`computeDeltas()` 클라이언트 계산**
   - 직전 동일 평형 거래 대비 증감% 계산. DB 쿼리 추가 없이 R2 데이터만으로 구현.

4. **Vercel 환경변수명 `NEXT_PUBLIC_R2_PUBLIC_URL` 사용**
   - 처음엔 `NEXT_PUBLIC_R2_URL`로 작성했다가, Vercel에 이미 `NEXT_PUBLIC_R2_PUBLIC_URL`이 설정돼 있어 변수명 일치시킴.

### 검토했지만 채택하지 않은 대안

| 대안 | 기각 이유 |
|---|---|
| R2 fetch를 TrendChart 내부 + ComplexInner 양쪽에서 별도 수행 | 동일 URL 이중 fetch. 브라우저 캐시로 커버되지만 구조적으로 불필요한 중복 |
| signals_mv 이력 유지하며 R2 전체 이력 별도 섹션 추가 | 화면 복잡도 증가. 현재 단계에서 시그널 배지보다 전체 이력이 더 가치 있음 |
| R2 데이터를 서버에서 집계 후 Supabase에 저장 | 매번 fetch_peaks 재실행 필요 + Route Handler 추가 → 무료 한도 영향 |
