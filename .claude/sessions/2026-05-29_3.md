## 의사결정 기록

### 채택한 결정

1. **전국 코드를 지식 기반으로 작성, API 실행으로 검증**
   - 이유: 행정표준코드관리시스템 실시간 크롤링 대신 훈련 지식으로 작성. 오류 코드는 `ingest.ts`가 로그 후 스킵하므로 데이터 안전성 보장. 실데이터 수집 시 자연 검증됨.

2. **날짜 변경 시 클라이언트 페치 (`/api/signals?date=`)**
   - 이유: ISR 제약 — `page.tsx`에서 `searchParams`를 읽으면 정적 생성이 깨짐. 최신 날짜는 ISR 캐시로 제공하고, 과거 날짜는 클라이언트가 동적으로 페치.
   - 불변 아키텍처 결정 #3(`일별 화면 ISR`) 유지: `/`의 ISR은 손대지 않음.

3. **recharts (MIT) 채택**
   - 이유: 사용자가 직접 선택. SVG 스파크라인(의존성 0) 대안도 검토했으나 인터랙티브 차트 요구. AGPL 아닌 MIT라 상업 이용 가능.

4. **시도→시군구 2단계 셀렉트 UI**
   - 이유: 250개를 flat 드롭다운에 넣으면 UX 불가. 시도 먼저 선택 후 해당 시도 시군구만 2번째 드롭다운 표시. URL 파라미터는 기존 `?region=` 유지(시군구 코드).

5. **`/api/signals` Cache-Control 헤더 추가**
   - 이유: 과거 날짜 데이터는 변하지 않으므로 `max-age=86400, stale-while-revalidate=604800` 적용. CDN/브라우저 캐시로 반복 요청 방어. Route Handler의 `revalidate` export는 동적 쿼리와 공존 어려워 response header 방식 채택.

### 검토했지만 채택하지 않은 대안

| 대안 | 기각 이유 |
|---|---|
| `/[date]` 동적 라우트 + `generateStaticParams` | 날짜가 매일 추가되므로 SSG 목록 관리 복잡. ISR Route 재구성 필요 → 기존 구조 유지 대비 이득 작음 |
| SVG 스파크라인(의존성 0) | 사용자가 recharts 선택 |
| 서버에서 시도 필터 처리 | ISR + searchParams 충돌(불변 아키텍처 결정 #3 위반) |
| `<optgroup>` 단일 셀렉트 (250개) | 250개 옵션을 스크롤하는 UX가 나쁨 |
