**#1 단지조회 드롭다운 뒤로가기 초기화**
- 시도/시군구/동을 선택할 때마다 URL params(`?sido=X&sgg=Y&umd=Z`)에 동기화
- `useState` 초기값을 `searchParams.get(...)` 으로 설정해 뒤로가기 시 URL → 상태 복원
- 핸들러 함수(`handleSidoChange`, `handleSggCdChange`, `handleUmdChange`)에서 `router.replace(..., { scroll: false })`로 URL 갱신

**#2 동 드롭다운 로딩 느림**
- 기존: 시군구 선택 → `/api/search?sgg=XXXXX` → 5컬럼 5000행 fetch → 클라이언트에서 umd 파생 → 느림
- 신규: `/api/search?sgg=XXXXX&fields=umds` → `umd_nm` 단일 컬럼 조회 → 서버 중복 제거 → 수십 개 문자열만 반환 → 빠름
- 동 선택 후에야 해당 동 단지 조회 (2단계 구조), apt dropdown은 동 선택 전 비활성화

## 의사결정 기록

### 동 목록 fast-fetch 방식 선택
- **채택**: `SELECT umd_nm FROM transactions WHERE sgg_cd=X AND canceled=false AND umd_nm IS NOT NULL LIMIT 5000` + 서버 측 JS 중복 제거
- **이유**: 스키마 변경(새 테이블/함수) 없이 기존 `/api/search` 라우트에 `?fields=umds` 분기만 추가. 응답 크기: 5컬럼 5000행(~100KB) → 50개 문자열(~500B)
- **불채택 대안 A**: Supabase RPC(`get_umds(sgg)`)로 `ARRAY_AGG(DISTINCT umd_nm)` → 가장 빠르나 마이그레이션 필요. 단순 UX 픽스에 DB 스키마 변경은 오버킬
- **불채택 대안 B**: 시군구 선택 시 기존 5000행 fetch를 유지하되 background fetch → umds는 파생, apts는 fetch 완료 후 → 큰 요청이 완료될 때까지 여전히 느림

### 뒤로가기 복원 — URL params vs sessionStorage
- **채택**: URL params에 상태 동기화 (`router.replace`)
- **이유**: 북마크·공유 가능, 세션 간 유지, isInitialMount 핵 없이도 clean
- **불채택**: sessionStorage — SSR 문제(typeof window check 필요), URL에 반영 안 됨

### 2단계 fetch UX 변경
- 기존 "동 선택 (전체)" 옵션으로 시군구 내 전체 단지 열람 가능했던 기능 제거
- 단, `sggCd` 선택 시 "X구 단지 전체 보기 →" 링크(`/complex/${sggCd}`)가 즉시 표시되어 기능 보상
- 현재 동 선택 없이 단지 드롭다운 = 비활성화 + "동 먼저 선택" 안내
