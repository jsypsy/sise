## 의사결정 기록

### 채택한 결정

1. **first_seen(등록일) 축 도입 + ingest 롤링 윈도우(당월+3개월)**
   - 이유(명세로 확정): 국토부 API(`RTMSDataSvcAptTradeDev`)는 요청 파라미터가
     `LAWD_CD`+`DEAL_YMD` 2개뿐이고, `DEAL_YMD`는 **"계약월"**. 등록일/신고일 기준
     조회 파라미터가 **아예 없음**. `rgstDate`는 "등기일자"라 신규 등록 추적용 부적합.
   - 신고 기한이 계약 후 30일이라, 오늘 등록되는 거래의 계약월은 전월·전전월일 수
     있음 → **당월만 긁으면 지연 신고분을 영구 누락**(reset 후 강동구 1건의 원인).
   - 표준 해법(호갱노노·아실·PublicDataReader 라이브러리 모두 동일): **최근 N개월을
     매일 다시 긁고, raw_key가 처음 나타난 날 = 등록일**으로 기록. 윈도우 폭은 신고기한
     30일이 2개 캘린더월에 걸치므로 **최소 2개월, 안전하게 3개월**.
   - KST 주의: Actions runner는 UTC라 `first_seen`을 DB `current_date`로 두면 cron
     시각(KST 04:10 = UTC 전날 19:10)에 하루 어긋남 → ingest가 **KST 날짜를 명시적으로**
     기록(런타임 검증 완료).

2. **archive cutoff를 계약일 → first_seen 기준으로**
   - 이유: 기존 archive가 "최근 20거래일(계약일) 이전"을 삭제 → 지연 신고된(계약일 오래된)
     거래가 들어와도 즉시 삭제됨. 등록일 기준으로 바꿔야 "등록 후 20일" 핫윈도우 유지.

3. **`sync_peaks_from_transactions()` daily 호출 제거** (신고가 self-inclusion 수정)
   - 진단(라이브 DB 샘플 쿼리): `price = prev_peak = peak_price, is_high=false`인 거래
     7건 확인 → 현재 거래 가격이 sync로 historical_peaks에 먼저 들어가 `prev_peak`에
     자기 값이 포함 → 진짜 신고가가 `is_high=false`로 억눌림.
   - sync는 archive와 역할이 겹침(archive도 떠나는 거래 peak을 hp에 병합) → 제거해도
     hp baseline은 archive + fetch_peaks(R2 전체이력)로 유지됨.

4. **커밋은 main 직접 + 푸시(Vercel 자동배포).** 기존 워크플로 유지.

### 검토했지만 채택하지 않은 대안

| 대안 | 기각 이유 |

| 지금 historical_peaks TRUNCATE 후 재구축 | 현재 transactions가 6월뿐이라 hp 비우면 단건 그룹이 전부 is_high=true → **신고가 폭주(거짓 양성)**. 더 나쁨 |
| fetch_peaks의 hp 범위 축소(당월−3)를 이번에 같이 | 사용자가 reload(6/6~6/13) 중 fetch-peaks 무변경 요청 → 6/13 후로 미룸 |
