## 의사결정 기록

### 채택한 결정

1. **온디맨드 재검증 (`revalidatePath`) vs Vercel Deploy Hook**
   - 채택: `POST /api/revalidate` + `revalidatePath("/", "layout")`
   - 이유: Deploy Hook은 전체 재빌드(~30초)를 트리거하지만, `revalidatePath`는 Next.js 캐시만 무효화하여 빌드 없이 즉시 갱신. 빌드 시간 절약 + Vercel Hobby 빌드 분(100h/월) 절약.
   - `REVALIDATE_SECRET` 환경변수로 인증 — 무단 캐시 무효화 방지.

2. **Node.js 22로 업그레이드**
   - 이유: `@supabase/realtime-js`가 Node.js 20에서 native WebSocket 없음 오류로 실패.
   - Node.js 22는 native WebSocket 지원 + 현재 LTS.
   - 향후 24가 Actions 기본값이 되더라도 22 → 24 마이그레이션 용이.

3. **Backfill 별도 워크플로우**
   - 이유: 일상 수집(`ingest.yml`)은 당월만. 소급 수집은 일회성 또는 필요 시 수동 실행. `workflow_dispatch` + 입력 파라미터로 유연하게 범위 지정 가능.
   - 불변 아키텍처 결정 #1(수집은 GitHub Actions) 준수.

4. **`useSearchParams` + `useEffect` 패턴 (complex page)**
   - 이유: URL 파라미터를 읽어 진입 시 자동 단지 로드. 기존 클라이언트 컴포넌트 구조 그대로 유지.
   - `useSearchParams`는 Suspense 경계 필요 → `ComplexInner` + `ComplexPage(Suspense 래퍼)` 패턴.

5. **Truncate 후 재수집**
   - Seed 데이터가 signals_v 계산(prev_peak/prev_price)을 오염시키므로 실서비스 전 truncate 필수.
   - Actions 실행 도중 truncate하면 partial 데이터 → Actions 먼저 취소 → truncate → 재실행 순서.
