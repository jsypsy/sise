### 추가된 기능 요약

1. **ingest 429 재시도**: fetchPage에 최대 4회 exponential backoff. 연속 5회 실패 시 조기 종료 (quota 소진 감지).

2. **fetch_peaks 429 재시도**: 동일한 backoff + `upsert_peaks_if_higher` RPC로 교체.

3. **historical_peaks GREATEST 전략**: 기존보다 낮은 값으로 덮어쓰는 버그 수정. `ON CONFLICT DO UPDATE WHERE EXCLUDED.peak_price > historical_peaks.peak_price`.

4. **ingest cron 앞당김**: KST 05:10 → 04:10. 실제 실행이 GitHub Actions 딜레이로 ~7:03이었음 → ~6:00 목표.

5. **bootstrap fetch-peaks 분리**: 일일 quota(10,000콜) 한계로 bootstrap에서 fetch-peaks Job 제거. `fetch_peaks.yml` 직접 실행으로 전환.

6. **fetch-peaks 5일 자동 배치**: 10,000콜/일 quota 내에서 3년(~9,000콜)씩 5일 자동 분할 실행.

7. **/complex PriceChart 제거**: 월별 추이 차트(TrendChart)만 표시. 일자별 차트 불필요.

## 의사결정 기록

### 채택한 결정

1. **ingest 연속 429 조기 종료 (5회 임계값)**
   - 이유: quota 소진 상태에서 250지역 × 30초 = 2시간 낭비. 5회 연속 실패 = quota 소진으로 판단, break.
   - 조기 종료 후에도 sync/archive/refresh는 정상 실행 (수집된 데이터 처리).

2. **historical_peaks GREATEST upsert**
   - 이유: fetch-peaks 배치가 과거 범위(예: 200601-200912)를 스캔할 때 현재 최고가(202605)를 낮은 값으로 덮어쓰는 버그 존재.
   - Supabase 클라이언트 `.upsert()`는 DO UPDATE 커스텀 불가 → SQL 함수로 해결.

3. **bootstrap에서 fetch-peaks 분리**
   - 이유: 244개월 × 250지역 = ~61,000 콜 필요하나 일일 quota 10,000. bootstrap에 포함 시 quota 소진 → 당일 ingest 실패.
   - fetch_peaks.yml(기존 워크플로우)을 수동 또는 자동 배치로 대체.

4. **fetch-peaks 5일 자동 배치 스케줄 (UTC 22:00)**
   - 이유: 매일 인트스(UTC 19:10, ~250콜) 완료 후 배치 실행. 합계 ~9,250콜 < 10,000 한도.
   - cron: 5월31일, 6월1-4일 각 UTC 22:00. 이후 날짜엔 exit 0.

### 검토했지만 채택하지 않은 대안

| 대안 | 기각 이유 |
|---|---|
| fetch-peaks를 bootstrap에 유지 | quota 일 10,000 초과로 당일 ingest 실패 |
| 배치 선택 드롭다운(수동) | 매일 클릭 필요. 자동화 요청으로 B안 채택 |
| fetch-peaks를 ingest보다 먼저(UTC 16:00) 실행 | GREATEST 버그 수정으로 순서 무관해짐 |
| 재시도 횟수 더 늘리기(>4회) | 총 대기 2+4+8+16=30초면 충분. 더 늘리면 한 지역에 1분+ 소비 |
